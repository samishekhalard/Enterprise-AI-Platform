# G02.04 Master License Management Database

**Track:** R02. TENANT MANAGEMENT  
**Technology:** PostgreSQL  
**Status:** MVP Review Ready  
**Supports:** `G01.02.02 Master License Management`

---

## 1. Purpose

This document defines the **authoritative master-license-management data model** used by `ADMIN (MASTER)` in the settings shell.

It supports:

- contracts inventory
- contract add and update
- contract end-date extension
- tenant license allocation overview
- expiry warnings
- renewal notifications
- restricted-access state after expiry

This model is distinct from the tenant-facing license read model stored in `Tenant Manager PostgreSQL Database.md`.

Use this separation:

- this document = authoritative contract and allocation model
- tenant manager PostgreSQL = tenant-facing projection and seat summary

## 2. Authoritative Tables

### 2.1 `license_contracts`

| Column | Type | Key / Rule | Purpose |
|--------|------|------------|---------|
| `contract_id` | `UUID` | PK | Contract identifier |
| `contract_number` | `VARCHAR(100)` | UNIQUE NOT NULL | Business contract number |
| `contract_holder_name` | `VARCHAR(255)` | NOT NULL | Contract holder |
| `contract_status` | `VARCHAR(30)` | NOT NULL | Draft, active, expiring, expired, suspended, or similar state |
| `start_date` | `DATE` | NOT NULL | Contract start date |
| `end_date` | `DATE` | NOT NULL | Contract end date |
| `renewal_state` | `VARCHAR(30)` | NULL | Renewal state |
| `created_at` | `TIMESTAMPTZ` | NOT NULL | Creation timestamp |
| `updated_at` | `TIMESTAMPTZ` | NOT NULL | Update timestamp |

Key relationships:

- `license_contracts` has many `license_contract_line_items`
- `license_contracts` has many `license_tenant_allocations`
- `license_contracts` has many `license_contract_extensions`
- `license_contracts` has many `license_expiry_alerts`
- `license_contracts` has many `license_notification_events`
- `license_notification_events` has many `license_notification_deliveries`
- `license_contracts` may have many `license_access_restriction_states`

---

## 3. Contract and Dashboard Rules

- one contract may allocate many license types
- one contract may allocate many tenants
- every tenant-allocation row belongs to exactly one contract
- contract extension is an operational event over an existing contract, not a replacement contract row
- expiry alerts are scheduled against contracts
- renewal notifications must support push, in-app, and email delivery channels
- restricted-access state after expiry is derived from the active contract and represented operationally through `license_access_restriction_states`
